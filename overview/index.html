Reference architecture

An open commerce network, ratified by a council, run on Beckn.

ION layers governance, a network profile, and three core services on top of the Beckn protocol — connecting buyer apps, providers, and consumers across Indonesian commerce.

The five layers, top to bottom

Governance → Profile → Services → Participants → User
L1 — Governance

ION Council

The ratifying body. Sets policies, approves changes to the network profile, and arbitrates disputes between participants.

L2 — Network profile

ION Network & registries

ion.yaml defines the network. The Policy Registry encodes what's allowed; the Error Registry standardises how participants report failure.

  • ion.yaml
  • Policy Registry
  • Error Registry
L3 — Service

ION Registry

The on-DeDi directory of every participant. BAPs and BPPs register here so the network knows who's authorised to transact.

  • on DeDi
L3 — Service

Catalogue Service

BPPs publish their offerings to the Beckn Fabric so buyer apps can search across providers without point-to-point integrations.

  • Beckn Fabric
L3 — Service

Discover Service (CDS)

BAPs query and subscribe to discover catalogues, providers, and updates relevant to their users.

  • queries
  • subscribes
L4 — Participant

BAP — Beckn Application Platform

The buyer-facing surface. Calls /select to express intent and consumes /on_select back from BPPs.

  • Tokopedia
  • Shopee
  • Lazada
  • Partner apps
L4 — Participant

BPP — Beckn Provider Platform

The supply-side surface. Merchants and logistics providers expose inventory, fulfilment, and order-state through standard endpoints.

  • Merchants
  • JNE
  • J&T
  • LSPs
  • Warehouses
Beckn Protocol Exchange — shared rail between every BAP and BPP: /init/confirm/status /track/update/reconcile
L5 — End user

Consumer

The person opening a buyer app. Everything above exists so a consumer in Jakarta can search a single storefront and transact with any registered provider on the network.
